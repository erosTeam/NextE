# 产品 Bug / Feature 工作规则

本文件负责产品语义、实现范围和用户路径验收。它不是任务队列；用户最新明确请求始终决定当前
工作，历史计划和证据只能用于定位。

## 开工前

1. 复述需要改变的结果和明确非目标，避免把问题替换成相邻问题。
2. 查当前 NextE 源码、相关提交和数据所有权；需要对标时给出 `eros_fe` 的具体文件/方法，平台
   表达再查 V2Next/HDS。
3. 写五行 grounding：参考实现、主信息、主/次动作、本轮可用闭环与非目标、HarmonyOS 表达方式。
4. 遇到会改变产品语义、数据字段/同步、信息层级或远端结构的取舍，列出收益与问题并等待用户
   决策，不通过绕过方案替用户决定。

## 实现纪律

- 修正确性回归时做改变行为的最小修复；不顺手重做 UI、parser、持久化或状态所有权。
- 设置项按用户心智归类；详情、列表、半模态、设置和管理页优先复用项目同类结构。
- 列表分页、刷新和缓存是不同层：查询必须有界，UI 增量加载，刷新保留可展示内容，失败不清空。
- 用户可见数据字段必须统一定义其语义、来源、存储、同步和展示位置；不能为了避开迁移而生成
  只在单设备成立的镜像/替代字段。
- EH 远端写入、账号和 Cookie 操作遵循 [EH integration contract](../eh-integration-contract.md)。

## 验收

- 先写一条真实用户路径：从哪里进入、做什么、状态如何变化、最终看到什么。
- 同时列出最容易坏的反例状态，例如加载中、空数据、缓存命中、长文本、小屏/折叠屏、键盘打开、
  滚动后、弹层切换或失败重试；选择与当前改动有关的至少三项。
- 日志证明动作/请求/状态链路，截图或 layout 证明可见结果，构建证明候选可安装。三者不能互相
  冒充。
- 没有运行时证据时写 `implemented candidate / needs QA`；已复现但仍失败写 `reopened / failed QA`；
  只有对应用户路径通过后才写 `accepted` 或 `completed`。
- 远端写入、资源消耗和不可逆动作默认验证到确认框或本地无害边界；真正提交必须有用户明确授权。

## 记录与生命周期

- 非平凡任务可在 `docs/plans/active/` 建一个当前计划；计划只记录当前范围、关键证据、未决项和
  完成判据，不建立跨任务调度队列。
- 实现完成且验证闭环后移到 `docs/plans/completed/`；被替代、长期停用、仅供查证或未接受但不再
  活跃的材料移到 `docs/plans/archive/`。
- 不维护全局追加式 grounding/事故/dispatch 账本。需要长期生效的规则合并到对应 agent guide；
  单次证据留在任务计划或 Git 历史。
