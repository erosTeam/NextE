# NextE 始终生效规则

这里只保留所有任务都适用的跨领域边界。ArkTS/UI、产品、EH、设备和本地构建细则由
[AGENTS.md](../../AGENTS.md) 分流到对应文档，不在这里重复。

## 范围与决策

- 用户最新明确请求决定当前任务。历史计划、归档、证据、handoff、租约和遗留工作区都不能自行
  创建任务、扩大范围或授权设备/远端操作。
- 可以自行处理不改变确认结果的可逆实现细节。涉及产品语义、数据归属、远端表结构、用户可见
  信息层级、不可逆行为或其他重大取舍时，先说明冲突、收益和代价，等待用户决策。
- 用户在任务中补充问题时，继续完成尚未取消的原目标；如果新请求明确替换旧目标，再停止旧目标。
- 不以“避免修改某层”为默认最高优先级。边界冲突必须按用户目标、数据一致性和当前架构共同评估，
  不能私自绕过后再让用户承担返工。

## 证据与完成边界

- 未知事实先变成证据。远端协议、页面 DOM、账号/资产/配额、数据同步、用户可见状态和平台 API
  不得靠猜测推进实现。
- 先查当前源码、提交历史和真实数据流，再改代码。参考项目提供语义或平台写法，不能替代 NextE
  当前实现证据。
- 构建、静态检查和代码存在只证明实现候选；设备截图、布局、日志、fixture 或真实用户路径才证明
  对应运行时行为。缺少黑盒证据时使用 `implemented candidate / needs QA`，不能宣称完成。
- 诊断任务只定位原因和证据；除非用户同时要求修复，不扩大为实现。
- 连续修补仍不能跑通时，停止叠加 workaround，回到状态所有权、数据流和最小可复现路径。

## Contract 使用边界

Contract 只保护会造成严重后果的稳定边界，例如：V2 状态管理、架构依赖、安全/凭据、远端写入、
同步与持久化数据完整性、真实 fixture 支撑的高风险协议/parser、设备互斥、版本/签名一致性。

不得为 UI 形状、布局、文案、视觉偏好、单次事故、临时 workaround、grounding 或普通验收流程新增
正则门禁。新增 contract 前必须能说明严重后果、现有验证为什么覆盖不了、失败后的修复方向；否则
使用源码审查、日志、截图、构建或设备路径验证。

## 仓库与提交

- 工作区已有改动属于用户。只改当前范围，保留无关 WIP；无法避开重叠时停下说明。
- 未经明确要求不提交、不推送、不建标签、不合并、不清理 worktree。
- 用户要求提交时，只暂存当前已确认并验证的路径；提交信息使用英文 Conventional Commits，敏感值
  不得进入提交信息、日志或 diff。
- 不使用破坏性 Git 命令覆盖用户改动。文档写着“已完成”不等于可以删除源码、文件或历史资产。

## 本地警示资料

`docs/agent-guides/incidents/` 只存本地开发警示，由 Git 忽略。受版本控制的文件不得链接、索引或
点名其中的具体文件，也不得 force-add 该目录。本地索引如有需要，只能保存在该忽略目录内部。

## 纠错输出

发生误改或规则违反时，不用口头保证替代行动。只报告：已经采取的具体动作、对应文件或规则、
验证结果和仍存在的阻塞。
